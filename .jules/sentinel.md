## 2025-02-28 - Preventing Thread Leaks in SSRF Checks
**Vulnerability:** When implementing SSRF protections requiring hostname resolution, using a simple thread pool to timeout `socket.getaddrinfo` can lead to thread exhaustion. Even if the timeout is caught, the underlying C-level network call continues to block the thread until OS timeouts occur. Also `0.0.0.0` is bypassable unless `is_unspecified` is handled.
**Learning:** `executor.shutdown(wait=False, cancel_futures=True)` helps garbage collect the executor, but it does not terminate blocked threads. The `cancel_futures` flag requires Python 3.9+. In a highly-concurrent environment, an attacker repeatedly hitting the timeout can DoS the app by exhausting available threads.
**Prevention:** Consider using fully asynchronous networking libraries (like `aiohttp` and `asyncio.get_event_loop().getaddrinfo()`) or separate isolated processes for hostname resolution to avoid risking the web server's main thread pool, or simply validating scheme and structure without synchronously resolving the IPs inline with every request. Also, `is_unspecified` needs to be accounted for.
