## 2026-06-21 - SSRF via Unvalidated URL Configurations
**Vulnerability:** User-configurable endpoints (`/api/aidr-config` and `/api/settings`) accepted URLs (`AIDR_BASE_URL`, `ollama_url`) without validation, allowing Server-Side Request Forgery (SSRF). An attacker could configure the application to hit internal network IPs or AWS Metadata services (169.254.169.254).
**Learning:** Configurable backend URLs that are hit by HTTP clients or instantiated by internal libraries must always be validated. The validation routine must actively resolve the hostname and block known private, link-local, multicast, and loopback IP spaces based on the operational context. For example, `ollama_url` was intended to support local usage so loopback/private IPs were safely allowed, while `AIDR_BASE_URL` needed strict blocking. Additionally, using `socket.getaddrinfo()` allows proper handling of both IPv4 and IPv6 resolution without DoS issues by running it inside a `ThreadPoolExecutor` with a timeout and ensuring it doesn't leak threads.
**Prevention:** Always parse untrusted URLs and perform IP resolution using `socket.getaddrinfo()` wrapped in an isolated thread with a hard timeout. Check the resolved IPs against known unsafe address ranges using Python's `ipaddress` module before attempting connections.
